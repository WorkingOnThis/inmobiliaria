# Flags contables en cargo manual + dialog de detalle de movimiento

**Fecha:** 2026-05-03  
**Estado:** aprobado

---

## Contexto

La tabla `tenant_ledger` ya tiene tres flags contables (`impactaPropietario`, `incluirEnBaseComision`, `impactaCaja`) y el endpoint POST ya los acepta. El problema: el dialog de cargo manual no los expone ni los envía — siempre usa los defaults del tipo. Tampoco existe forma de ver o editar los flags de un movimiento existente.

---

## Alcance

1. **Dialog de cargo manual**: agregar los tres flags, pre-cargados con los defaults del tipo seleccionado.
2. **Dialog de detalle de movimiento**: accesible desde el `...` de cualquier fila. Editable para entradas manuales, solo lectura para auto-generadas.
3. **LedgerTable**: consolidar acciones en `...` para todos los rows; eliminar la X de punitorios.

Los dos componentes nuevos viven en `src/components/ledger/` para ser reutilizables en cuenta corriente de propietarios y caja.

---

## Sección 1 — API

### PATCH `/api/tenants/[id]/ledger/[entryId]`

Agregar al `patchSchema`:
```typescript
impactaPropietario: z.boolean().optional(),
incluirEnBaseComision: z.boolean().optional(),
impactaCaja: z.boolean().optional(),
```

Agregar guard de seguridad: leer `isAutoGenerated` del entry existente. Si es `true`, rechazar cualquier cambio a `monto`, `descripcion`, `dueDate` o los tres flags con `422 Unprocessable Entity`. Solo permitir `estado` y `cancellationReason` en entradas auto-generadas.

### GET `/api/tenants/[id]/cuenta-corriente`

Incluir en cada entry devuelta:
- `isAutoGenerated: boolean`
- `impactaPropietario: boolean`
- `incluirEnBaseComision: boolean`
- `impactaCaja: boolean`

## Sección 2 — Tipos y estructura de archivos

Agregar cuatro campos:

```typescript
impactaCaja: boolean
```

### Archivos nuevos


### `AddManualChargeDialog`

```typescript
type ManualChargeData = {
  tipo: "gasto" | "servicio" | "bonificacion" | "descuento"
  descripcion: string
  monto: number
  period?: string
  impactaPropietario: boolean
  incluirEnBaseComision: boolean
  impactaCaja: boolean
}

type Props = {
  open: boolean
  onOpenChange: (open: boolean) => void
  onSave: (data: ManualChargeData) => Promise<void>
}
```

- Maneja todo su estado de formulario internamente.
- Cuando `tipo` cambia, los flags se resetean a `defaultFlagsForTipo(tipo)` (importado de `@/lib/ledger/flags`).
- Flags UI: tres `Switch` de shadcn con labels en español.

### `EntryDetailDialog`

```typescript
type EntryEditData = {
  descripcion?: string
  monto?: number
  dueDate?: string
  impactaPropietario?: boolean
  incluirEnBaseComision?: boolean
  impactaCaja?: boolean
}

type Props = {
  entry: LedgerEntry | null   // null = cerrado
  onOpenChange: (open: boolean) => void
  onSave: (data: EntryEditData) => Promise<void>
}
```

- `entry.isAutoGenerated === true` → todos los campos deshabilitados (vista de solo lectura).
- `entry.isAutoGenerated === false` → campos editables: descripcion, monto, dueDate, y los tres flags.
- Los flags se muestran en ambos casos.
- El botón "Guardar" solo aparece cuando `isAutoGenerated === false`.
- `onSave` es provisto por el padre — el componente no conoce URLs ni inquilinoId.

## Sección 3 — LedgerTable

### Props que cambian

| Acción | Detalle |
|---|---|
| Se agrega | `onViewDetail: (entry: LedgerEntry) => void` |
| Se elimina | `onCancelPunitorio: (id: string) => void` — se reemplaza por `onCancelEntry` genérico |

`onCancelEntry: (entry: LedgerEntry) => void` ya existe. El padre recibe la entry completa y decide qué mutación usar según `entry.tipo === "punitorio"` (DELETE directo, sin dialog de motivo) o cualquier otro tipo (PATCH a `cancelado`, abre dialog de motivo).

### Comportamiento de acciones por fila

Todos los rows tienen `...` menu. La X de punitorios desaparece.

| Tipo de entry | Opciones en `...` |
|---|---|
| Cualquiera | "Ver detalle" → `onViewDetail(entry)` |
| Punitorio no conciliado | + "Cancelar punitorio" → `onCancelEntry(entry)` |
| No punitorio, cancelable | + "Cancelar movimiento" → `onCancelEntry(entry)` |

El recibo (número + anular) sigue fuera del menú, sin cambios.

> El `...` de punitorios usa el mismo tamaño que los demás rows para facilitar el toque.

## Sección 4 — `tenant-tab-current-account.tsx`

### Estado que sale

Todo el bloque de estado del cargo manual (`showManual`, `manualTipo`, `manualDescripcion`, `manualMonto`, `manualPeriod`, `manualError`) se mueve al componente `AddManualChargeDialog`.

### Estado que entra

```typescript
const [selectedDetailEntry, setSelectedDetailEntry] = useState<LedgerEntry | null>(null)
```

### Mutación que entra

`editEntryMutation`: PATCH a `/api/tenants/:id/ledger/:entryId` con los campos editados. Se pasa como `onSave` al `EntryDetailDialog`.

### Resultado

El archivo pierde ~120 líneas de JSX (dialog inline) y queda como coordinador: provee callbacks con contexto (inquilinoId, contratoId, propietarioId, propiedadId) y delega el render del formulario a los componentes de `ledger/`.

## Fuera de alcance

- Edición de flags en cuenta corriente de propietarios y caja — los componentes quedan listos para eso, pero los endpoints y la integración se hacen en una tarea separada.
- Pool de motivos de cancelación (ya en PENDIENTES.md como prioridad baja).
- Edición de `period` en el dialog de detalle — demasiado riesgoso sin lógica de re-generación de cuotas; se descarta por ahora.
