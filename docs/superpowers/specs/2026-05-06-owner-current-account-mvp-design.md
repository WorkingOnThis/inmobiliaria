# Diseño — Cuenta corriente de propietarios (MVP)

**Fecha:** 2026-05-06  
**Estado:** Aprobado  
**Scope:** Vista de cuenta corriente en la ficha del propietario

---

## Contexto

La ficha del propietario ya tiene una tab "Cuenta corriente" con un componente
`OwnerTabCurrentAccount`. Ese componente existe pero está incompleto:

- El menú `···` de cada fila tiene "Ver detalle", pero el handler es un no-op.
- No hay `EntryDetailDialog` montado.
- No hay filtros (se muestran todas las entradas sin posibilidad de filtrar).
- Solo muestra un KPI (`totalLiquidadoYTD`) en un card de texto plano.

El objetivo es cerrar esa brecha para que la vista sea funcional como herramienta
de consulta del staff.

---

## Qué NO está en scope

- Cancelar movimientos desde la vista del propietario (los entries del owner CC son
  filas del `tenantLedger` compartidas con el inquilino; cancelar desde aquí afectaría
  ambas vistas y requiere una política contable que todavía no está definida).
- Editar entries desde la vista del propietario (requiere un endpoint nuevo que no
  existe aún — `PATCH /api/owners/:id/ledger/:entryId`).
- Préstamos a propietarios (feature futuro que requiere diseño de schema separado,
  probablemente una tabla `owner_ledger`).
- Filtro de lista de propietarios por estado de liquidación (feature separado).

---

## Cambios — 4 archivos, ninguno nuevo

### 1. `src/components/ledger/entry-detail-dialog.tsx`

Agregar prop `readOnly?: boolean`. Cuando es `true`, fuerza `isEditable = false`
sin importar si la entrada es manual o auto-generada.

```ts
// Antes
const isEditable = entry !== null && !entry.isAutoGenerated && ...

// Después
const isEditable = !readOnly && entry !== null && !entry.isAutoGenerated && ...
```

El botón "Guardar" solo aparece cuando `isEditable = true`, por lo que en modo
`readOnly` el dialog es solo de lectura automáticamente.

---

### 2. `src/components/tenants/ledger-table.tsx`

Usar el prop `isOwnerView` (ya existe) para ocultar las opciones destructivas del
menú `···`:

- "Cancelar punitorio" → oculto cuando `isOwnerView = true`
- "Cancelar movimiento" → oculto cuando `isOwnerView = true`
- "Anular recibo" → oculto cuando `isOwnerView = true`
- "Ver detalle" → siempre visible

---

### 3. `src/app/api/owners/[id]/cuenta-corriente/route.ts`

Agregar una tercera consulta paralela que calcule el total pendiente de liquidar:
suma de `monto` donde `propietarioId = id AND impactaPropietario = true AND estado
IN ('pendiente', 'registrado')`.

Respuesta actualizada:

```json
{
  "kpis": {
    "totalLiquidadoYTD": 125000,

  },
  "ledgerEntries": [...]

```

### 4. `src/components/owners/owner-tab-current-account.tsx`

Tres mejoras al componente:

**KPIs (reemplaza el card único):**
- Card "Liquidado YTD" → monto en color `--income`, etiqueta con el año actual.
- Card "Pendiente" → si `totalPendiente > 0` muestra monto en `text-warning`;
  si es 0 muestra "Al día" en `text-income`.

**Filtros:**
- `ToggleGroup` de tipo `multiple` con tres opciones:
  - `pending` → "Pendientes"
  - `paid` → "Pagados"

**EntryDetailDialog:**
- Nuevo estado `selectedDetailEntry: LedgerEntry | null`.

- `EntryDetailDialog` montado con `readOnly={true}` y `onSave` como no-op
  (`() => Promise.resolve()`).

---

## Decisión de diseño: readOnly en el dialog

El `EntryDetailDialog` ya tiene lógica de `isEditable` basada en `isAutoGenerated`
y `estado`. La mayoría de las entradas en la CC del propietario son auto-generadas
(alquiler mensual, ajuste de índice), por lo que ya serían read-only naturalmente.

Forzamos `readOnly={true}` igualmente porque:
1. Las entradas manuales (cargos de staff) podrían aparecer como editables en el
   dialog, pero no tenemos endpoint para guardar desde el lado del propietario.
2. Es más seguro forzar read-only ahora y habilitarlo cuando esté el endpoint.

---

## Notas para el futuro

- Cuando se implemente edición desde la vista del propietario: crear
  `PATCH /api/owners/:id/ledger/:entryId` y quitar `readOnly={true}`.
- Cuando se implemente la feature de préstamos: el componente `OwnerTabCurrentAccount`
  ya recibe un array de `LedgerEntry` — se puede extender para combinar entries de
  `tenantLedger` con entries de una futura tabla `owner_ledger` en la API.
- El filtro "liquidación pendiente" en la lista de propietarios es una feature
  separada anotada en PENDIENTES.md.
